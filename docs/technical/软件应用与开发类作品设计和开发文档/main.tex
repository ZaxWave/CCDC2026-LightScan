\section{需求分析}                                                                                                                                                 
  \subsection{开发背景}                                                                                       
  中国公路网已全面进入"建养并重"的存量维护时代。
  据交通运输部《2024年交通运输行业发展统计公报》\cite{mottstat2024}，
  2024年末全国公路总里程达549万公里，高速公路19.07万公里，
  均居世界第一；2025年公路养护市场规模达1.25万亿元，
  交通基础设施巡检平台市场规模预计突破800亿元，
  年复合增长率达18\%--20\%。

  政策层面持续加码。《“十四五”公路养护管理发展纲要》\cite{shiswu2022}
  明确提出路面自动化采集覆盖率达100\%的目标；
  2025年9月，交通运输部等七部门联合印发
  《关于"人工智能＋交通运输"的实施意见》\cite{aiplus2025}，
  在"智能建养"部分明确提出"推广应用轻量化智能路面检测车"，
  "推进人工智能技术在交通基础设施日常巡查、检测监测、
  病害诊断、技术状况评定、养护决策等应用"，
  为路面缺陷AI智能检测的落地提供了明确的顶层支撑。

  然而现实存在显著落差。
  《公路技术状况评定标准》（JTG\,5210-2018）\cite{jtg52102018}规定高速公路
  每年检测两次、普通国省干线一至两次，
  但专业检测车单台采购价200--350万元，
  检测外包费1000--1200元/公里·车道，
  全国公路养护资金缺口约50\%，大量路段实际检测频次严重不足，
  病害从萌芽到恶化的演变过程长期处于监测盲区。

  更深层的矛盾在于\textbf{数据来源的结构性匮乏}。
  现有方案均以专用检测设备为唯一采集入口，
  而每日在公路上高频往返的公交车、环卫车、出租车、邮政车
  已内置行车记录仪，无人机、道路监控摄像头亦广泛部署，
  市民与巡检人员随时携带智能手机——
  这些\textbf{存量数据资产}因缺乏统一接入与处理能力而长期沉睡。
  多源设备在坐标系、时间戳、图像质量上存在异构性，
  \textbf{多源数据的融合与空间对齐}是将其转化为可用巡检数据的核心技术挑战。

近年来，随着计算机视觉技术的飞速发展，深度学习在路面病害检测领域已取得显著进展，
但在复杂光照、遮挡等极端环境下的鲁棒性仍是当前学术界的研究热点 \cite{fan2024survey}\cite{lin2026yolo11wlbs}\cite{aqeel2025roadfusion}\cite{cheng2025survey}。
传统的检测方法往往依赖于单一传感器，而本项目则尝试在轻量化架构下实现更高精度的突破。

  现有方案聚焦"精确检测"，共同缺陷是硬件门槛高、
  数据跨次孤立、检测结果与工单管理割裂。
  LightScan 定位于\textbf{无专用硬件条件下的高频次、可追踪、有闭环常态化巡检}，
  填补县乡级管养单位的技术空白。
  % ── 新增句 ────────────────────────────────────────────
  系统支持手机拍照、行车记录仪视频（公交车、无人机、私家车等）、
  巡检员上传与市民众包等多元采集入口，
  通过时空约束与视觉 ReID 融合算法实现跨源数据的统一对齐与去重聚类，
  将存量公共车辆与市民资源转化为常态化巡检网络。

  \subsection{用户需求}

  根据上述背景，系统面向三类角色的核心诉求如下：

  \begin{itemize}
      \item \textbf{市民}：发现路面问题后需要一个\textbf{零门槛的上报渠道}，
      无需注册、无需专业判断，10 秒内完成拍照提交即可。

      \item \textbf{基层巡检员}：日常巡检中\textbf{不应额外采购设备}，
      手机与行车记录仪应能直接作为采集工具；
      病害识别与定位应自动化，减少人工判读负担；
      工单分派、处理与核销需有清晰的流转记录。

      \item \textbf{养护管理部门}：需掌握辖区病害的\textbf{空间全貌与演变趋势}，
      而非仅有离散的单次检测快照；
      需要从"发现病害"到"修复核验"的\textbf{闭环管理}，
      确保每条工单有人负责、有结果可查；
      高频次的多源数据汇聚应能支撑\textbf{预防性养护决策}，
      在病害恶化前介入。
  \end{itemize}

由此归纳四项核心需求：
  \begin{enumerate}[label=\textbf{R\arabic*.}, leftmargin=3.5em, itemsep=2pt]
    \item \textbf{多源异构数据统一接入}：手机、行车记录仪、无人机及市民
          众包数据的空间对齐与标准化处理；
    \item \textbf{轻量化实时识别}：无专用硬件依赖的检测引擎，
          降低技术与经济门槛；
    \item \textbf{时空演变追踪}：跨批次、跨来源的病害点自动聚合，
          支持病害生命周期全过程监测；
    \item \textbf{全生命周期闭环}：覆盖病害发现、工单分发到修复验收
          的数字化管理流程。
  \end{enumerate}

  \subsubsection{视觉规范与原子化设计}

  系统 Web 端采用工业级深色主题设计语言，以 \texttt{\#0d1117}（背景）、
  \texttt{\#3E6AE1}（品牌蓝）、\texttt{\#39ff14}（告警绿）为核心色板，
  强调信息密度与专业感，与城市基础设施管理场景相契合。
  移动端（Taro 4）沿用同一色系，针对触控操作将可点击目标高度提升至 44 dp 以上，
  并以角色卡片（市民 / 巡检员）替代传统菜单导航，降低非专业用户的认知负担。

  原子化组件层面，Web 端抽取了 \texttt{UploadArea}、\texttt{ProgressBar}、
  \texttt{StatsRow}、\texttt{ResultsGrid} 等通用组件，通过 Props 驱动复用于
  图片检测和视频检测两个面板；地图层拆分为 \texttt{ClusterLayer}、
  \texttt{HeatmapControls}、\texttt{TimelineModal} 三个独立子组件，
  各自封装 AMap 生命周期，便于独立维护。

\subsubsection{反范式设计说明}
本系统在 \texttt{disease\_records} 表中冗余存储了本应属于 \texttt{disease\_clusters} 表的 \texttt{label\_cn} 字段，并直接存储了高维特征指纹 \texttt{feature\_vector}。该违背第三范式的设计基于以下理由：

\begin{enumerate}[label=\arabic*.]
    \item \textbf{查询性能优化}：在 GIS 态势感知大屏渲染万级病害散点时，若通过多表关联（JOIN）获取病害类型信息，会产生巨大的计算开销与 I/O 延迟。通过冗余存储，系统可实现单表索引覆盖查询，确保前端地图加载与缩放的极致流畅感。
    \item \textbf{聚类算法响应速度}：基于 ReID 的视觉聚类算法需要极高频地比对历史特征向量。将特征指纹直接物理存储在观测记录表中，避免了跨表读取产生的 I/O 损耗，使实时检测引擎的聚类响应速度能够满足工业级道路巡检的实时性要求。
    \item \textbf{数据快照一致性}：道路病害随时间具有演变性。反范式存储能够完整记录采集瞬时的视觉特征与分类状态，为后续进行病害恶化趋势的对比分析提供不可篡改的原始”快照”数据。
\end{enumerate}

\paragraph{大字段分离策略}
\texttt{repaired\_image\_b64} 与 \texttt{thumbnail\_b64} 以 Base64 文本直接存入主表会导致数据库体积膨胀 30\% 以上，严重拖慢全表扫描与索引性能。为此，系统引入独立的 \texttt{disease\_media} 表（见表~\ref{tab:disease_media}），将图片的存储路径（OSS/MinIO URL）与主表解耦：列表类查询仅读取主表的元数据字段，不再加载大块 Base64 文本；详情页按需从 \texttt{disease\_media} 获取图片 URL 或兜底 Base64。主表现有 Base64 字段保留以维持向后接口兼容。

 \subsubsection{LS-Det 核心检测算法研发}       
                                                                                  
  LS-Det 是团队基于轻量化卷积神经网络架构，针对道路病害检测场景自主研发的检测模型。在骨干网络设计上，参考学术界 YOLO11s 的先验设计\cite{lin2026yolo11wlbs}\cite{ultralytics2024}，加载预训练权重，进行参数调整和网络重构，采用深度可分离卷积替代标准
  卷积，引入多尺度特征融合机制（FPN+PAN）\cite{zhang2024steeldefect}，在保持实时推理速度的同时兼顾小目标裂缝的检测精度。检测头针对道路病害的长条形、不规则
  形态特征进行了解耦优化，显著降低误检率。

  \textbf{训练方案：}模型在公开数据集 RDD2022 \cite{arya2022rdd2022}\cite{alam2025crowd} 上训练，经过 180 个 epoch，采用 960×960 输入分辨率、设置 Patience=40 的早停策略以防止过拟合、box\_loss 权重 7.5 的自定义损失配置，最优权重于第 108 epoch 收敛（RTX 4060，训练耗时约 10 h）。

% --- 算法 1：SAHI ---
\subsubsection{基于 SAHI 的自适应多尺度增强检测}

\textbf{设计难点}：行车记录仪及监控影像分辨率不固定。当原始图像直接缩放至检测器标准尺寸（$640\times640$）时，细微病害（如像素宽度 $<10$\,px 的裂缝）的纹理特征损失严重。在有限的边缘端算力下，如何既保证小目标的高召回率，又避免因全分辨率推理导致的实时性下降，是本算法的核心挑战。

\textbf{技术方案}：引入 \textbf{SAHI (Slicing Aided Hyper Inference)} 框架\cite{akyon2022sahi}，通过滑动窗口对原图进行自适应切片。系统根据输入分辨率 $R_{\text{in}}$ 动态调整窗口尺寸 $S_{\text{slice}}$ 与重叠率 $r_{\text{overlap}}$：

\begin{equation}
(S_{\text{slice}},\; r_{\text{overlap}}) = 
\begin{cases} 
(640\times640,\; 0\%) & \text{if } R_{\text{in}} \leq 640 \\
(512\times512,\; 20\%) & \text{if } 640 < R_{\text{in}} \leq 1280 \\
(640\times640,\; 15\%) & \text{if } R_{\text{in}} > 1280 
\end{cases}
\end{equation}

通过设置合理的重叠率确保了跨切片病害的完整性。子图独立推理后，经 NMS 算法合并至原图坐标系，整体流程无需重新训练模型即可显著提升小目标检测性能。

% --- 算法 2：融合聚类 ---
\subsubsection{时空约束与视觉 ReID 融合聚类算法}

\textbf{设计难点}：车载 GPS 定位在复杂路况下常存在 3--15\,m 的随机漂移，单纯依赖空间坐标会导致同一病害产生多条冗余档案。如何在采集数据流中，利用病害自身的“视觉指纹”作为唯一标识来修正低精度的空间坐标，实现精准的“一病一档”，是系统逻辑设计的难点。

\textbf{技术方案}：系统研发了“空间+视觉”双门限判别模型，并结合 DBSCAN 空间聚类算法\cite{ester1996dbscan}对重复病害进行剔除。首先计算新记录与候选簇间的空间得分 $S_{\text{spatial}}$。同时，提取检测框内 32 维归一化 HSV 特征向量，计算其与候选记录的余弦相似度 $V_{\text{visual}}$。系统依据视觉相似度动态调节空间合并阈值 $d_{\text{threshold}}$：

\begin{table}[H]
\centering
\zihao{6}
\caption{视觉相似度与空间门限动态映射关系}
\begin{tabular}{ccc}
\toprule
余弦相似度 $V_{\text{visual}}$ & 语义匹配判定 & 空间合并阈值 $d_{\text{threshold}}$ \\ \midrule
$\geq 0.92$ & 强视觉匹配 & 8\,m (容忍较大 GPS 漂移) \\
$[0.75,\; 0.92)$ & 中等匹配 & 5\,m (标准地理门限) \\
$< 0.75$ & 视觉差异过大 & 拒绝合并 (判定为独立新实体) \\ \bottomrule
\end{tabular}
\end{table}

\textbf{融合判别式}：$\text{Score} = 0.35 \times S_{\text{spatial}} + 0.65 \times V_{\text{visual}}$。当 $\text{Score} \geq 0.62$ 时判定为同一病害实体并赋予统一的 \texttt{cluster\_id}。若图像质量极差无法提取特征，系统将退化至纯空间判断模式。

\textbf{PostGIS 空间索引加速}：在 PostgreSQL + PostGIS 环境下，候选簇搜索从经纬度矩形包围盒升级为 \texttt{ST\_DWithin(...::geography, 8\,m)} 球面距离查询，借助 GIST 空间索引将大规模点位的候选检索时间复杂度从 $O(n)$ 降至 $O(\log n)$，同时消除高纬度地区矩形包围盒的投影畸变误差。无 PostGIS 时系统自动降级为原包围盒策略，核心聚类逻辑不受影响。新建聚类时系统还将同步将置信度与检测框面积转化为 1--5 级病害严重度（\texttt{severity}），作为工单优先级排序的初始依据。

% --- 算法 3：演变趋势 ---
\subsubsection{基于时序观测的病害演变趋势分析算法}

\textbf{设计难点}：在无相机标定的众包影像中，病害的物理尺寸难以精确测量。如何在异构设备拍摄的像素面积波动中提取出稳健的病害恶化趋势，是养护决策支持的关键。

\textbf{技术方案}：系统对同一 \texttt{cluster\_id} 记录进行 90 天窗口内的时序分析。
提取检测置信度 $C$ 与检测框相对像素面积 $A$。通过计算变化斜率 $k$ 进行状态预警：
\begin{equation}
\text{Status} = 
\begin{cases} 
\text{Deteriorating (恶化)} & \text{if } \Delta C > 0.05 \text{ and } \Delta A > \delta \\
\text{Improving (改善)} & \text{if } \Delta C < -0.05 \\
\text{Stable (稳定)} & \text{others}
\end{cases}
\end{equation}

该算法通过相对趋势而非绝对尺寸实现分级预警，为养护优先级排序提供数据支撑。为了消除设备俯仰角变化引入的透视形变误差，系统引入了直方图特征的一致性核验，确保演变判定是基于病害区域的质变而非单纯的像素波动。通过这种相对趋势分析，系统能够对“带病运行”的路段进行恶化风险分级，从而辅助管理部门在有限的预算内精准锁定制订预防性养护优先级。

% --- 算法 4：OCR 采样 ---
\subsubsection{基于视频 OCR 的离散里程采样算法}

\textbf{设计难点}：在不依赖高精定位硬件的前提下，仅凭视频画面实现物理空间上的“等距采样”难度极高。停滞时产生的重复帧与高速行驶时的采样空隙，是造成云端数据冗余与漏检盲区的根因。

\textbf{技术方案}：利用 \textbf{PaddleOCR} \cite{paddleocr2020}定向识别视频流中烧录的车速叠字。系统每隔 15 帧（约 0.5\,s）解析一次瞬时车速 $v_i$，通过离散步进累算行驶里程 $\Delta S$：
\begin{equation}
\Delta S = \sum_{i} \left( \frac{v_i}{3.6} \cdot \Delta t_i \right), \quad \Delta t_i = \frac{15}{f_{\text{fps}}}
\end{equation}

当累积位移达到预设采样间距（默认 10\,m）时触发推理引擎。此外，系统内置了异常速度过滤机制，当 OCR 识别出现置信度波动的瞬时跳变时，利用前序帧的平均加速度进行线性插值补偿，确保了里程累计的连续性。这种“以语义特征补偿物理传感”的设计，使得系统在车辆频繁启停的城市工况下，仍能保持极高的空间对齐精度与数据采集能效。

\subsection{安装过程}
本节详细说明后端、Web 前端及移动端的安装步骤。

\subsubsection{后端安装}

\noindent\textbf{步骤 1}：创建并激活 Python 虚拟环境：
\begin{lstlisting}[language=bash]
conda create -n lightscan python=3.11 -y
conda activate lightscan
\end{lstlisting}

\noindent\textbf{步骤 2}：安装 PyTorch（按显卡型号二选一，已使用续行符处理长命令）：
\begin{lstlisting}[language=bash]
# RTX 50 系列 Blackwell 架构驱动 (Nightly)
pip install --pre torch torchvision \
    --index-url https://download.pytorch.org/whl/nightly/cu128

# RTX 40 系列及以下架构
pip install torch torchvision \
    --index-url https://download.pytorch.org/whl/cu124
\end{lstlisting}

\noindent\textbf{步骤 3}：安装其余依赖并初始化数据库：
\begin{lstlisting}[language=bash]
pip install -r requirements.txt
createdb lightscan
\end{lstlisting}

\noindent\textbf{步骤 4}：在 \texttt{src/backend/} 目录下新建 \texttt{.env} 文件（注意下划线转义）：
\begin{lstlisting}[language=bash]
DATABASE_URL=postgresql://用户名:密码@localhost:5432/lightscan
SECRET_KEY=your-secret-key
\end{lstlisting}

\noindent\textbf{步骤 5}：启动后端服务：
\begin{lstlisting}[language=bash]
cd src/backend
uvicorn app.main:app --reload --port 8000
\end{lstlisting}

\subsubsection{Web 前端安装}

\noindent\textbf{步骤 1}：在 \texttt{src/frontend/} 下新建 \texttt{.env}，填入高德地图密钥：
\begin{lstlisting}[language=bash]
VITE_AMAP_KEY=你的高德Key
VITE_AMAP_SECURITY_CODE=你的安全密钥
\end{lstlisting}

\noindent\textbf{步骤 2}：安装依赖并启动：
\begin{lstlisting}[language=bash]
cd src/frontend && npm install && npm run dev
\end{lstlisting}
完成后通过浏览器访问 \texttt{http://localhost:5173}。

\subsubsection{移动端安装}
\begin{lstlisting}[language=bash]
cd src/mobile && npm install

# H5 模式预览
npm run dev:h5        # 访问 http://localhost:10086

# 微信小程序端编译
npm run build:weapp   # 用微信开发者工具导入 dist/weapp/
\end{lstlisting}

\subsection{典型使用流程}

\subsubsection{路面图片检测入库}
\begin{enumerate}[label=\arabic*., nosep]
    \item \textbf{影像上传}：用户在 Web 端批量拖拽上传现场病害照片（最多 20 张）。
    \item \textbf{智能识别}：系统调用 LS-Det 引擎自动标注病害类型并提取视觉特征。
    \item \textbf{自动归档}：识别结果经时空聚类后自动写入数据库并在 GIS 地图实时落位。
\end{enumerate}

\subsubsection{行车记录仪视频巡检}
\begin{enumerate}[label=\arabic*., nosep]
    \item \textbf{视频解析}：上传 MP4 视频，系统通过 OCR 识别车速或接收预设均速。
    \item \textbf{等距采样}：算法按设定里程（如 10m）自动抽帧，执行高性能切片推理。
    \item \textbf{批量入库}：用户核验汇总结果后，一键完成整条路段的病害批量建档。
\end{enumerate}

\subsubsection{多源移动端上报流程}
\begin{enumerate}[label=\arabic*., nosep]
    \item \textbf{众包采集}：市民或巡检员通过小程序选择病害类型并拍摄现场照片。
    \item \textbf{位置对齐}：系统自动触发高精度 GPS 采集，并将坐标与影像信息关联。
    \item \textbf{实时上报}：点击提交后，数据即刻汇入后台工单池，实现发现即上报。
\end{enumerate}

\subsubsection{工单管理与闭环}
\begin{enumerate}[label=\arabic*., nosep]
    \item \textbf{派单}：管理员在 GIS 大屏点击病害点，指派任务至对应巡检员。
    \item \textbf{修复}：巡检员通过移动端接收待办，完成修复后上传对比照片。
    \item \textbf{对比验收}：修复完成后，系统开放卷帘式 Before-After Slider 对比界面，管理员拖动滑块实时比对修复前后影像，确认质量后完成验收闭环。
    \item \textbf{归档}：系统核验修复影像，GIS 散点状态同步变更为”已修复”并进入时间轴归档。
\end{enumerate}

\section{项目总结}

  \subsection{开发历程与团队协作}

  本项目从需求调研到系统上线，历经行业调研、技术选型、
  算法设计、前后端联调与移动端适配五个阶段。
  团队按模块分工：算法与后端、Web 前端、移动端并行推进，
  通过版本管理与接口约定保持协同。
  实践中深刻体会到"接口先行"的重要性——
  前后端同时开发时，提前约定 JSON 数据结构与错误码规范，
  显著减少了联调返工成本。

  \subsection{攻克的主要难点}

  \textbf{多源数据对齐}是本项目最具挑战性的工程问题。
  不同设备的 GPS 精度差异可达 3--15\,m，
  单靠坐标无法可靠判断两次拍摄是否为同一病害点。
  为此设计了时空约束与视觉 ReID 融合的双门限聚类算法，
  以 32 维 HSV 特征向量作为视觉指纹补偿 GPS 漂移，
  最终在测试集上实现了准确的跨源病害点归并。

  \textbf{移动端跨平台适配}同样耗费了大量调试精力。
  Taro 在微信小程序环境下对 CSS 规范存在诸多限制，
  通配符选择器、\texttt{scroll-view} 内边距、
  \texttt{verbatim} 类组件嵌套等均需特殊处理；
  微信开发者工具灰度基础库的已知兼容性问题，
  最终通过真机预览绕过，积累了宝贵的跨端工程经验。

  \subsection{后续演进方向}

  \begin{enumerate}[label=\arabic*., itemsep=2pt]
      \item \textbf{端侧推理}：将 LS-Det 量化为 INT8 格式，
      部署至移动端本地，实现离线检测，摆脱网络依赖。

      \item \textbf{B 端车队接入}：与公交、环卫等运营车辆系统对接，
      将车载行车记录仪视频流自动化接入巡检管道，
      形成"千辆车即千个巡检站"的城市级覆盖网络。

      \item \textbf{严重度量化与预测性养护}：
      当前已实现基于 ls-det 置信度与检测框面积的 1--5 级自动初评（\texttt{severity} / \texttt{priority} 字段），
      并结合路段等级支持人工覆写。后续将引入裂缝宽度估算与坑槽体积计算，
      结合时间轴演变数据训练病害恶化预测模型，
      自动生成精细化养护优先级排序与维修建议报告。

      \item \textbf{商业化路径}：
      以 SaaS 模式面向公路管养单位提供轻量化订阅服务，
      契合交通运输部推广"轻量化智能检测"的政策导向，
      目标市场为公路养护部门。
  \end{enumerate}
